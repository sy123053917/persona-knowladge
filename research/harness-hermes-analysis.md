# Harness 与 Hermes Agent 深度研究报告

> **研究对象**：Harness Engineering（AI Agent工程范式）与 Hermes Agent（自进化Agent框架）
>
> **研究时间**：2026年4月22日

---

## 一、纵向分析：两个时代命题的诞生与演进

### 1.1 Harness Engineering：AI Agent 的「驯兽师」哲学

#### 起源：从「聪明的模型」到「可靠的 Agent」

2026年之前，AI Agent 领域存在一个根本性的悖论：大语言模型在单轮对话中表现出令人惊叹的智能，但一旦被置于需要多步骤执行、长时间运行的生产环境中，就会暴露出系统性的脆弱性。

模型会遗忘指令、执行顺序错乱、工具调用失败后无法恢复、在复杂任务中陷入无限循环。研究者们逐渐意识到：**问题的根源不在于模型本身太弱，而在于缺乏一个能够「驾驭」模型的工程框架。**

这个认知催生了「Harness Engineering」这一概念。

#### 诞生节点：2026年2月，OpenAI 发布标志性技术博客

2026年2月，OpenAI 发布了一篇名为《Harness Engineering: Leveraging Codex in an Agent-First World》的技术博客。这篇文章的核心洞见是：AI Agent 的可靠性瓶颈不在于模型能力，而在于**工程化程度不足**。

Harness（缰绳/套具）这个名字本身就透露着设计哲学：模型是野性的力量，而 Harness 是让这股力量被安全、可控地驾驭在特定任务中的工程框架。

博客中披露了一个引人深思的实验：一个仅由 Codex 驱动的 Agent，在没有额外工程干预的情况下执行真实编程任务时，成功率不足 40%。但当加入了完善的 Harness 层——包括记忆管理、任务拆解、错误恢复、安全审批——之后，同一个 Agent 的成功率跃升至 92% 以上。

这个实验成为整个行业的分水岭时刻。

#### 演进历程：三条主线的交织

**第一条主线：Claude Code 的意外「被开源」**

Anthropic 的命令行编码工具 Claude Code 是 Harness Engineering 最早的标杆实践。2025年3月31日，一个意外的疏漏让外界获得了其超过51万行 TypeScript 源码的访问权——npm 发布包中的 source map 文件意外暴露了存储在 R2 存储桶中的未混淆源码。

这份泄露的源码揭示了一个令业界震惊的事实：Claude Code 拥有一个极为精密的 Harness 系统，远比外界此前猜测的「增强版 CLI」复杂得多。

其核心架构包含五大支柱：

1. **工具集（Tools）**：为编程任务量身定制的文件系统操作、终端命令执行、Git 操作、代码搜索等工具，每个工具都有完善的参数校验和错误处理。
2. **上下文管理器（Context Management）**：包括系统提示构建、工作目录分析、对话历史压缩、关键文件内容注入等多层策略。
3. **安全审批层（Approval Layer）**：所有潜在危险操作（文件删除、网络请求、Shell 命令执行等）必须经过用户审批才能执行，审批规则可配置。
4. **会话管理系统（Session Management）**：跨会话维护任务状态，支持任务中断和恢复。
5. **资源治理（Resource Governance）**：Token 预算控制、超时管理、并发限制。

**第二条主线：范式的正式确立**

2026年2月之后，Harness Engineering 迅速从 Claude Code 的实现细节演变为整个 AI Agent 领域的工程范式。OpenAI、Anthropic、Google DeepMind 等主要玩家纷纷发布相关技术文档，将 Agent 的 Harness 层视为与模型层同等重要的基础设施。

这一趋势在2026年3月达到一个小高潮：Anthropic 正式发布 Claude Code 的官方 Harness 文档，详细披露了其四大核心能力——安全沙箱、长运行会话、多 Agent 协调、可信治理。Notion、Sentry、Asana 等企业率先采用，报告称开发效率提升高达10倍。

**第三条主线：OpenClaw 的 ACP 协议**

在开源生态中，OpenClaw 的 ACP（Agent Client Protocol）协议代表了 Harness Engineering 的另一种实践路径。ACP 不是一个具体的产品，而是一个**编排与执行的通信协议标准**。

通过 ACP 协议，OpenClaw 可以将 Claude Code、Qwen Code、Gemini Code 等外部编码工具作为「一级子代理」来调度，实现跨模型的无缝协作。用户向 OpenClaw 发出请求，OpenClaw 通过 ACP 协议分发给最适合的编码 Agent 执行，执行结果再通过 ACP 协议返回给 OpenClaw 进行整合。

这种架构被称为「Hub-and-Spoke」模型：OpenClaw 是中枢大脑，通过 ACP 协议连接各种专业化的编码 Agent。Hub 负责理解意图、全局规划、记忆管理、质量把控；Spoke（子代理）负责执行特定类型的任务。

#### 决策逻辑：为什么需要 Harness？

在每一个关键节点，Harness Engineering 的演进都遵循同一个核心逻辑：**让不确定性变成确定性。**

大语言模型的核心特征是非确定性——同样的输入可能产生不同的输出。这在聊天场景中是可以接受的（甚至是有趣的），但在需要精确执行的 Agent 场景中，这是不可接受的。

Harness 层的设计目标就是在不改变模型本身的前提下，通过工程手段将模型的非确定性约束在可控范围内：

- 模型可能「忘记」关键指令 → Harness 用系统提示和上下文注入来反复强化
- 模型可能选择错误的工具 → Harness 用工具描述和示例来引导
- 模型可能在失败后放弃 → Harness 用重试策略和备选方案来兜底
- 模型可能执行危险操作 → Harness 用审批层来把关

这不是在限制模型的能力，而是在**为模型提供一个安全、可预测的工作环境**。

### 1.2 Hermes Agent：「会自我进化的数字伙伴」的诞生

#### 起源：Agent 的「失忆症」困境

几乎同一时期，另一个根本性的问题引起了不同团队的关注：传统 AI Agent 每次对话结束后，对这次任务学到的一切——解决问题的方法、用户的偏好、成功的策略——全部归零。

这形成了一个令人沮丧的循环：

```
传统 Agent：
  任务 A → 解决 → 结束 → 遗忘
  任务 B → 解决 → 结束 → 遗忘
  ...
  每次都要从零开始，每次都是「新的」Agent
```

用户为此付出了大量重复性的「提示词劳动」：每次开新会话都要重新设定自己的偏好、解释项目背景、纠正 Agent 的重复错误。这被社区形象地称为「提示词 PTSD」。

#### 诞生节点：2026年2月，Nous Research 的破局之作

面对这个问题，美国开源 AI 实验室 **Nous Research**（以 Hermes 系列大模型闻名）于2026年2月下旬发布了 **Hermes Agent**。

这个名字取自希腊神话中的信使之神，暗示着项目的核心追求：**让知识在 Agent 的使用过程中不断沉淀和传承，而非每次会话后消散。**

与大多数 Agent 框架不同，Hermes Agent 从一开始就被定位为「自我改进型 Agent」（Self-Improving Agent）。这不是一个简单的功能特性，而是整个系统的设计核心。

**v0.8.0（2026年4月8日）** 是第一个达到周更迭代的稳定版本，社区活跃度极高。

#### 演进历程：45天，6万 Stars 的爆发式增长

**第一阶段：发布期（2026年2月下旬）**

项目以 MIT 协议完全开源，核心特性是三层记忆架构和闭环技能学习系统。上线第一周便获得 8,000+ GitHub Stars，在 AI Agent 分类中迅速攀升至第一。

**第二阶段：社区验证期（2026年3月）**

技术社区开始深度使用并分享体验。一个核心话题引发广泛讨论：Hermes Agent 在长期任务中展现出的「进化感」——用户发现，Agent 在第三、第四次处理同类任务时，明显比第一次更快、更准确。

Reddit 上的一个高赞评论写道：「10/10，相当于已经调试了一周的 OpenClaw + RAG + 记忆持久化 + 更好的工具调用。」

**第三阶段：爆发期（2026年4月）**

项目 GitHub Stars 突破 60,000（45天），超越 OpenClaw 的增长曲线。浙商证券等券商发布行业研究报告，将 Hermes Agent 定义为「Agent 时代」的核心标的之一。

值得关注的是，Hermes Agent 与 OpenClaw 并不是简单的竞争关系。Hermes Agent 官方内置了 `hermes claw migrate` 命令，支持一键迁移 OpenClaw 的设置、记忆、技能和 API 密钥。两者在用户群体中存在大量重叠——许多用户同时使用两个框架，用 OpenClaw 做多平台执行，用 Hermes 做长期记忆和技能积累。

#### 技术架构：五层记忆与闭环学习

Hermes Agent 的技术实现是其竞争力的核心。其记忆系统采用了一个精心设计的五层架构：

| 层级 | 类型 | 持久性 | 说明 |
|------|------|--------|------|
| L1 | Transformer 上下文 | 会话内 | 短期推理，会话结束即消失 |
| L2 | SKILL.md 程序性知识 | 永久 | 技能沉淀，Agent 的「肌肉记忆」 |
| L3 | 向量存储索引 | 永久 | 语义检索，支持模糊召回 |
| L4 | Honcho 辩证式用户建模 | 持续进化 | 对用户的理解随每次对话更新 |
| L5 | FTS5 全文检索 + LLM 摘要 | 永久 | 跨会话搜索历史对话 |

其中最独特的是 **L4 层的 Honcho 用户建模引擎**。Honcho 借鉴了黑格尔的「正-反-合」辩证法：

- **正题（Thesis）**：当前对用户的理解（画像现状）
- **反题（Antithesis）**：新对话中出现与当前理解矛盾或补充的信息
- **合题（Synthesis）**：LLM 将正题和反题融合，生成更新的用户理解

这意味着用户说过的偏好——「我喜欢简短的要点式答案」「我的代码风格用制表符不用空格」——不会被遗忘，而是被持续整合进用户画像中。

**闭环学习的实际效果**体现在技能系统上。当 Agent 完成一个任务后，会自动将执行流程沉淀为 SKILL.md 文件。下次遇到类似任务时，直接调用该技能。执行后发现问题？自动更新技能文件。

```
创建技能 → 使用技能 → 发现改进点 → 更新技能 → 下次使用更好的版本
```

这形成了一个正反馈循环，Agent 的能力随使用时间指数级提升。

#### RL 训练飞轮：最独特的差异化

Hermes Agent 最独特的差异化在于它内置了**强化学习（RL）训练数据飞轮**：

```
[Agent 执行任务] → [生成轨迹数据] → [压缩清洗] → [RL 训练] → [更强模型] → 回到起点
```

这是因为 Nous Research 同时拥有 Atropos（RL 环境框架）和 Tinker（RL 训练器），Hermes Agent 是第一个将 Agent 框架与 RL 训练框架完全打通的系统。

用户使用得越多，训练数据质量越高，下一代模型越强。这形成了一个其他竞品难以复制的壁垒。

## 二、横向分析：两个维度的竞品对比

### 2.1 竞品格局判断

**场景B：存在直接竞品，但各自代表不同维度**

需要明确的是，Harness Engineering 和 Hermes Agent 并不在同一个维度上直接竞争：

- **Harness Engineering** 是一个**工程范式/方法论**，代表 AI Agent 构建的系统性方法论
- **Hermes Agent** 是一个**具体的产品实现**，是 Harness Engineering 哲学的忠实实践者

真正的竞品关系存在于：

| 维度 | 主要竞品 |
|------|---------|
| Harness Engineering（范式） | OpenClaw ACP、LangChain Agents、AutoGPT |
| Hermes Agent（产品） | OpenClaw、Claude Code、DeerFlow |

### 2.2 核心差异对比

#### 技术路线对比

| 维度 | Harness Engineering | Hermes Agent | OpenClaw |
|------|---------------------|--------------|----------|
| **核心范式** | 工程框架方法论 | 闭环学习系统 | Hub-Spoke 编排 |
| **记忆机制** | 依赖实现者设计 | 五层记忆架构（含用户建模） | 记忆文件（MEMORY.md） |
| **技能系统** | 视具体实现而定 | 主动创建与优化 | 手动/Skill 框架 |
| **自我进化** | 不内置 | ✅ 闭环学习 | 有限（Self-Improving 技能） |
| **RL 训练集成** | 不内置 | ✅ Atropos 飞轮 | ❌ |
| **多平台支持** | 视具体实现 | ✅ 12个平台 | ✅ 20+平台 |
| **模型灵活性** | 依赖集成 | ✅ 200+模型 | ✅ 多模型 |

#### 定位与哲学对比

**Harness Engineering（以 Claude Code 为代表）**的哲学是「可靠执行」。

Claude Code 的设计目标是成为一个**工业级的编程助手**。它的每一个功能都围绕着同一个目标：让 AI 在生产级编程任务中变得可预测、可审计、安全可控。

代价是灵活性相对较低——Claude Code 主要针对编码场景优化，不太适合做通用任务。

**Hermes Agent** 的哲学是「持续成长」。

Hermes 的设计目标是成为一个**会随用户一起成长的数字伙伴**。它的核心价值不在于单次任务完成率，而在于长期积累的能力复利。

代价是短期内的「慢启动」——一个新部署的 Hermes 需要经过若干次交互才能展现出真正的价值。

**OpenClaw** 的哲学是「全栈集成」。

OpenClaw 的设计目标是成为一个**多平台、多场景的 AI 自动化中枢**。它最强大的地方在于丰富的插件生态和跨平台消息集成。

代价是记忆和技能系统相对 Hermes 较为被动——需要用户主动维护或借助额外的技能框架。

#### 用户口碑对比

**Claude Code / Harness 的典型评价**：

- 「它让我第一次真正相信 AI 可以帮我完成真实的生产代码。」
- 「上下文管理太精准了，不需要反复提醒它我项目的结构。」
- 「审批系统稍微繁琐，但用久了确实安心。」

**Hermes Agent 的典型评价**：

- 「第三周开始，我意识到它真的在『记住』我——不只是我说的，还有我做事的方式。」
- 「技能自动生成这个功能太香了，终于不用自己写 CLAUDE.md 了。」
- 「在树莓派级别的设备上也能跑，资源占用确实优化得很好。」

**OpenClaw 的典型评价**：

- 「飞书、微信、钉钉全接进来之后，我的工作流自动化程度直接提升了一个量级。」
- 「Skill 框架很强大，但维护起来需要投入不少精力。」
- 「生态太丰富了，刚入门的时候有点不知所措。」

### 2.3 生态位分析

三个系统在 AI Agent 版图中占据着截然不同的生态位：

**Claude Code（及 Harness Engineering）**占据的是「专业编码助手」的生态位。这是目前最成熟、商业化程度最高的 AI Agent 细分市场。目标用户是专业开发者，核心价值是提升编程效率。

**Hermes Agent**占据的是「个人 AI 伙伴」的生态位。这是一个相对蓝海的市场——大多数 AI Agent 框架关注的是「完成任务」，而 Hermes 关注的是「与用户一起成长」。目标用户是愿意深度定制、长期使用的个人用户或研究者。

**OpenClaw**占据的是「AI 自动化中枢」的生态位。它的核心价值不是单次任务完成，而是跨平台、跨场景的自动化编排。目标用户是需要将 AI 集成到现有工作流程中的用户。

## 三、横纵交汇：在 AI Agent 演进坐标系中定位未来

### 3.1 两个时代命题的交汇

Harness Engineering 和 Hermes Agent 实际上在回答同一个根本问题：**如何让 AI Agent 从「工具」进化为「伙伴」？**

Harness Engineering 从工程维度回答了这个问题：通过对齐、安全沙箱、上下文管理、审批机制等工程手段，将模型的不确定性约束在可控范围内。这解决的是「如何让 Agent 可靠地执行」的问题。

Hermes Agent 从认知维度回答了这个问题：通过多层记忆、闭环学习、用户建模等技术，让 Agent 真正理解用户并持续积累经验。这解决的是「如何让 Agent 有记忆地成长」的问题。

两个维度并不矛盾，而是互补的。

一个真正理想的 Agent 框架，应该同时具备：

1. **Harness 层的可靠性**——安全、可控、可审计
2. **Hermes 的成长性**——记忆用户、积累技能、持续进化

这也是为什么许多用户同时使用 OpenClaw（提供 Harness 层的工程可靠性）和 Hermes Agent（提供认知层的成长性）的根本原因。

### 3.2 趋势判断

**趋势一：Harness 与 Hermes 的融合将加速**

未来的 Agent 框架将越来越难以将「工程可靠性」和「认知成长性」割裂开来。用户不会满足于「安全但健忘」的 Agent，也不会满足于「聪明但危险」的 Agent。

我们已经看到这一趋势的端倪：OpenClaw 正在通过 Self-Improving 技能框架引入主动学习机制；Hermes Agent 在 v0.9 中也在强化其安全审批层。两个框架正在互相「渗透」对方最强的能力域。

**趋势二：标准化之战即将到来**

Harness Engineering 的兴起催生了对 Agent 互操作标准的强烈需求。ACP 协议、MCP（Model Context Protocol）、agentskills.io 等标准正在争夺生态主导权。

谁将成为 AI Agent 世界的「USB-C」？这个问题的答案将决定未来 5 年的生态格局。

**趋势三：Personal Agent 的时代正在开启**

Hermes Agent 90天内获得的关注度（以及大量用户选择同时运行 OpenClaw 和 Hermes）表明，一个新的需求正在崛起：**用户希望拥有一个真正属于自己的、越用越懂自己的 AI Agent，而非每次都要重新「入职培训」的通用助手。**

这不是一个技术问题，而是一个体验范式的转变。当用户开始说「我的 Hermes」而非「Hermes」的时候，Personal Agent 时代就真正到来了。

### 3.3 机会与风险

| 维度 | 机会 | 风险 |
|------|------|------|
| **Harness Engineering** | 成为行业标准的基础；所有 Agent 框架都将采纳 | 过度工程化导致灵活性下降；标准碎片化 |
| **Hermes Agent** | 占据 Personal Agent 蓝海；RL 数据飞轮形成壁垒 | 依赖 Nous Research 的持续投入；短期商业化路径不清晰 |
| **整体生态** | 两者融合催生新一代 Agent 框架 | 安全与成长的平衡仍需探索；监管风险上升 |

---

## 结语

Harness Engineering 和 Hermes Agent，一个回答了「如何让 AI 可靠」，一个回答了「如何让 AI 成长」。

它们不是竞争对手，而是同一场 Agent 革命的两种叙事。

一个像造车：精密的工程、严格的安全标准、可靠的系统架构。
一个像养马：耐心的陪伴、持续的成长、越来越默契的配合。
